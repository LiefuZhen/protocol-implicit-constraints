# 隐式约束生成方向提示词 v0

用途：本文件是阶段 I 最终产物的草案位置。正式版本应由 T3 的 CVE 溯因、聚类和人工核实结果蒸馏得到。当前 v0 仅作为 T1/T6 阶段的格式占位和后续迭代基础。

## 输入

你将收到一份目标协议标准文本或其中的结构化片段，包括章节、字段、消息、状态机、术语定义、错误处理和安全考虑。

## 任务

请基于“生成方向”定位标准中可能隐藏隐式约束的位置，并为每个位置生成候选隐式约束。方向关注的是标准缺陷形态和反推思路，而非漏洞危害分类。

## 生成方向

1. **沉默/欠定义处**
   - 定位：标准规定正常取值、正常路径或常见输入，但对边界、越界、异常、极值、畸形输入处理缺少清楚说明。
   - 反推：若接受异常情形会破坏协议目标，则候选约束应要求显式拒绝、有界处理或一致归一化。

2. **前后矛盾 / 交叉引用不一致**
   - 定位：同一字段、状态、能力或行为在不同章节中出现范围、强度、优先级或术语含义不一致。
   - 反推：实现必须保持一致解释；当不同解释会影响安全或互操作时，候选约束应采用更能保持协议目标的解释。

3. **能力/范围声明含糊**
   - 定位：标准使用 allow、support、accept、extend 等表述，但未量化支持程度、长度、数量、嵌套深度、版本跨度或异常处理。
   - 反推：实现接受或声明支持某能力时，应保持该能力相关的协议目标；不支持的输入应显式拒绝或清楚降级。

4. **目的与机制脱节**
   - 定位：标准说明字段或机制用于唯一标识、请求响应关联、防重放、完整性、防降级、缓存隔离等目的，但未写出完整实现步骤。
   - 反推：从该目的反推出最小必要行为；违反后会破坏明确协议目标的约束标为 L1。

## 输出格式

对每个候选位置输出：

```json
{
  "direction_id": "",
  "spec_ref": {
    "standard": "",
    "section": "",
    "entity": "",
    "quote_or_summary": ""
  },
  "spec_defect_form": "",
  "statement": "",
  "condition": "",
  "expected_behavior": "",
  "violation_pattern": "",
  "bound_goal": "",
  "strength": "L1_or_L2",
  "review_notes": ""
}
```

## 复核要求

- 每条候选约束必须绑定标准位置和协议目标。
- L1 表示违反后会破坏明确协议目标；L2 表示鲁棒性、最佳实践或较弱合规要求。
- 对标准本身边界不清的情况标记为 `ambiguous_spec` 候选。
- 输出中保留证据链，便于进入候选约束 JSON 和后续实现验证。
