# 隐式约束生成方向说明

本文件提供 T1/T6 阶段的种子版方向说明。正式方向集将在 T3 阶段基于已复核 CVE 的溯因和聚类形成，并蒸馏为阶段 I 的最终提示词 artifact。

## 1. 方向的含义

本项目中的“方向”指的是：

> 在协议标准的哪种模糊、欠定义、前后矛盾或目的未落地位置下手，以及如何从该位置反推出隐式约束。

方向关注“去哪找、怎么反推”，候选约束关注“什么条件下应有什么行为”。两者关系如下：

```text
CVE 根因
  -> 标准缺陷形态
  -> 生成方向
  -> 目标标准中的可疑位置
  -> 候选隐式约束
```

## 2. 种子方向示例

下表用于说明一条方向的基本形态。正式方向名称、数量和边界由 T3 的真实 CVE 聚类结果确定。

| Direction ID | 中文名 | 针对的规范缺陷形态 | 在标准中如何定位 | 反推隐式约束的思路 |
|---|---|---|---|---|
| `boundary_silence` | 沉默/欠定义处 | 标准只规定正常路径，对边界、越界、异常、极值处理沉默。 | 找“只说正常行为、缺少异常分支”的条款。 | 若异常输入会破坏协议目标，则要求显式拒绝、有界处理或一致归一化。 |
| `cross_reference_conflict` | 前后矛盾/交叉引用不一致 | 同一字段、状态或行为在多处描述不一致。 | 找重复定义、交叉引用、术语漂移和范围冲突。 | 实现必须保持一致解释；必要时采用更能保持协议目标的解释。 |
| `ambiguous_capability_scope` | 能力/范围声明含糊 | 标准允许或支持某能力，但未界定支持程度。 | 找 allow/support/accept/extend 等无量化边界表述。 | 接受或声明支持时应完整保持相关语义；无法支持时应显式拒绝或清楚降级。 |
| `purpose_mechanism_gap` | 目的与机制脱节 | 标准说明目的，但未把目的落实为完整实现步骤。 | 找“用于唯一标识/关联/防重放/完整性/防降级”等目的性描述。 | 从目的反推出最小必要行为，并绑定明确协议目标。 |

## 3. 当前协议示例

| 协议 | 方向 | 候选约束示例 |
|---|---|---|
| MQTT | `purpose_mechanism_gap` | 服务端接受长 ClientId 时，应保持身份完整性，避免静默截断导致身份坍缩。 |
| MQTT | `ambiguous_capability_scope` | 支持超过兼容长度的 ClientId 时，应完整保留身份语义或显式拒绝。 |
| CoAP | `purpose_mechanism_gap` | Token 应保持请求-响应关联，避免截断或跨 endpoint 错配。 |
| DNS | `boundary_silence` | compression pointer 解析应保证包内、有界、无环。 |
| DNS | `cross_reference_conflict` | 缓存和权威边界相关条款出现冲突时，应采用保持 bailiwick 安全目标的解释。 |

## 4. L1/L2 强度

| 强度 | 判断标准 | 例子 |
|---|---|---|
| L1 | 违反后会破坏明确协议目标。 | ClientId 静默截断破坏身份唯一性；DNS pointer 循环破坏解析终止性。 |
| L2 | 偏向经验规则、鲁棒性建议或最佳实践。 | 错误码选择、日志行为、超时策略。 |

L1 可进入实现缺陷判定流程；L2 更适合作为可疑点或辅助证据。

## 5. T3 方向归纳要求

T3 阶段需要从 T2 接受的隐式 CVE 中重新归纳方向：

1. 对每条 CVE 做 evidence-bounded 溯因，记录标准缺陷形态；
2. 按“规范缺陷形态 + 反推思路”聚类；
3. 人工核实每个簇的适用边界；
4. 形成正式 direction set v0；
5. 将方向集蒸馏为 `prompts/distilled_direction_prompt.v0.md` 的正式版本；
6. 报告 seed 方向的保留、合并、拆分或淘汰结论。
